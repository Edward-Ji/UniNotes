Binary Search Trees

A binary search tree (BST) is a data structure for storing values in a sorted manner.

These values are laid out so that an in-order traversal of the BST visits the values in ascending order.

A more formal definition would be that for any node vvv﻿ in the tree, any node uuu﻿ in the left subtree of vvv﻿, and any www﻿ in the right subtree of vvv﻿, key(u)<key(v)<key(w)\def\key{\text{key}} \key(u) < \key(v) < \key(w)key(u)<key(v)<key(w)﻿.

This definition of a binary search tree does not allow duplicate keys. However, we can:

  • Allow the left descendants to be equal to the parent.
  • Using a list to store duplicates.

However, it means additional complexity in the BST implementation.

Operations

Search

To search for a key kkk﻿ with a BST, we

  • Start at the root;
  • Compare kkk﻿ with the stored value at each node.
    • If kkk﻿ is smaller than the stored value, we go left;
    • If kkk﻿ is equal to the stored value, we found the key;
    • If kkk﻿ is greater than the stored value, we go right.
  • If we reach an external node, the key is not in the tree.

Analysis

Binary tree searching runs in O(h)O(h)O(h)﻿ time, where hhh﻿ is the height of the tree.

In the worst case, we can take h=n−1h = n-1h=n−1﻿. The algorithm runs in O(n)O(n)O(n)﻿ time.

In the best case, we can balance a BST so that the height of the tree h=⌈log⁡2n⌉h = \lceil\log_2 n\rceilh=⌈log2​n⌉﻿. The algorithm then runs in O(log⁡n)O(\log n)O(logn)﻿ time.

Variation

Database systems use a generalization of BST to index data and allow for fast lookups. Two key features of external memory complicate the task:

  • External memory is orders of magnitude slower than internal memory.
  • External memory is transferred in blocks of size BBB﻿ (order of 10310^3103﻿ bytes).

Our goal is to minimize the number of transfers (i.e. I/O operations).

If we implement BST as is in external memory, we need log⁡2n\log_2 nlog2​n﻿ I/Os even if the tree is perfectly balanced. When nnn﻿ is very large and the throughput of queries is high, this becomes too slow, so database systems resort to a different data structure that subdivides the tree furthermore.

Insertion

To perform operation put(k, o), we search for key k using the previous search method. If k is found in the tree, replace the corresponding value with o. If k is not found, let w be the external node reached by the search. We replace w with an internal node holding (k, o).

Deletion

To perform operation remove(k), we search for key k using the previous search method to find the node w holding k. We distinguish between two cases:

  • If w has one internal child, we replace w with the only child of w.
  • If w has two internal children, we find the internal node y following w in an in-order traversal. We then copy the entry from y into node w and demote it to an external node.

    Note that y is also the smallest value in the right subtree of w, and the immediate value after w if sorted in ascending order. The algorithm would also work if we take the largest node in the left subtree of w.

If k is not in the tree, we can either throw an exception or do nothing.

Analysis

get, put and remove all runs in O(h)O(h)O(h)﻿ time, where hhh﻿ is the height of the tree.

In the worst case, we can take h=n−1h = n-1h=n−1﻿. The algorithm runs in O(n)O(n)O(n)﻿ time.

We can balance the tree by using a better insertion routine so that h=⌈log⁡2n⌉h = \lceil\log_2 n\rceilh=⌈log2​n⌉﻿. The algorithm then runs in O(log⁡n)O(\log n)O(logn)﻿ time.

Balanced Binary Search Trees

Motivation

The simple insertion implementation described above (insertion in sorted order) could lead to a tree with h=n−1h=n-1h=n−1﻿. We can balance the binary search tree with simple local transformations so that the height hhh﻿ is O(log⁡n)O(\log n)O(logn)﻿. This translates to an O(log⁡n)O(\log n)O(logn)﻿ searching algorithm.

AVL Trees

AVL Trees are rank-balanced trees. It satisfies the constraint that the ranks of the two children of every internal node differ by at most 111﻿ where the rank r(v)r(v)r(v)﻿ of node vvv﻿ is the height of the subtree rooted at vvv﻿.

Insertion/Deletion

To insert kkk﻿ into an AVL tree, we first search for kkk﻿ in the tree.

  • If kkk﻿ is in the tree, the search ends at the node holding kkk﻿. There is nothing to change.
  • If kkk﻿ is not in the tree, search for kkk﻿ ends at external node xxx﻿. Add kkk﻿ as a child of xxx﻿.

The new node containing kkk﻿ may not be of O(log⁡n)O(\log n)O(logn)﻿ depth, so we need to apply rotation to the tree. Let www﻿ be the new node containing kkk﻿, and x,y,zx, y, zx,y,z﻿ be nodes on the path from www﻿ to root. Knowing the height of subtrees of x,y,zx, y, zx,y,z﻿, we can rearrange the three nodes so that the new tree is still an AVL tree.

This operation requires the height of each node. If we were to compute this by recursively computing the height of each subtree, it would take O(n)O(n)O(n)﻿ time. Therefore, we need to store this as an attribute of each node and update this after each insertion and rotation operation. This way the insertion only runs in O(h)=O(log⁡n)O(h) = O(\log n)O(h)=O(logn)﻿ time.

Deletion works in a similar fashion.

List-Based Implementation

We can implement an unsorted map using a list of key-item pairs.

The get(k) and put(k,v) operations would both take O(n)O(n)O(n)﻿ time.

This is only feasible if the number of operations or the data quantity is small.

Tree-Based Implementation

We can implement a sorted map directly using an AVL tree, where each node stores a key-item pair.

To perform a get or put, we search for the key in the tree, so these operations take O(log⁡n)O(\log n)O(logn)﻿ time.

This is only feasible if there is a total ordering on the keys.

Skip Lists

This is yet another way to implement a map. It is a relatively simple data structure that still has O(log⁡n)O(\log n)O(logn)﻿ average-case time complexity.

It is a leveled structure, where every level is a subset of the one below it. The elements of the upper level are determined randomly and linked to the same element below.

Each node ppp﻿ has after(p), before(p), above(p), and below(p) functions to access the node to the left, to the right, on the level above, and on the level below respectively.

Applications

  • Database systems
  • Concurrent/Parallel programming

Operations

Search

To search for kkk﻿ in a skip list, at each level,

  • if kkk﻿ is greater than the value of the current node, move right;
  • if kkk﻿ is smaller than the value of the current node, move left;
  • if kkk﻿ is equal to the value of the current node, move down.

Until we are already at the bottom level, the node is returned if it contains kkk﻿, otherwise, kkk﻿ is not found.

Insertion

To insert kkk﻿ in a skip list, we perform the search operation and insert it at that position.

Then, we determine randomly if we are to add the node to the levels above. At each level, it has a half chance to also be added to the next level.

Deletion

To delete kkk﻿ in a skip list, we perform the search operation and remove it from the bottom level.

Then, we keep removing it from the level above as long as it exists at that level.
