Three Abstractions

Computational problem

  • defines a computational task
  • specifies what the input is and what the output should be

Algorithm

  • a step-by-step recipe to go from input to output

    (what your solution does)

  • different from implementation

Correctness and complexity analysis

  • a formal proof that the algorithm solves the problem

    (why your solution is correct)

  • analytical bound on the resources it uses

Efficiency

The definition should be implementation and context independent. It should:

  • count number of “steps”
  • bound the algorithm’s worst-case performance
  • be objective

Definition

An algorithm is efficient if it runs in polynomial time; that is, on any instance of size nnn﻿, it performs no more than p(n)p(n)p(n)﻿ steps for some polynomial p(x)=adxd+⋯+a1x+a0p(x) = a_dx^d + \dots + a_1x + a_0p(x)=ad​xd+⋯+a1​x+a0​﻿.

This gives us information about the worst-case behavior of the algorithm, which is useful for making predictions and comparing different algorithms.

What operations take constant time?

Definition

Constant time O(1)O(1)O(1)﻿ means running time does not depend on the size of the input.

Examples

  • Assignments a <- 42
  • Comparisons =, <, >
  • Boolean operations and, or, not
  • Basic arithmetic operations +, -, *, /
  • Constant sized combinations of the above, e.g. a <- (2*b + c)/4

Naive Algorithm

This algorithm uses the accumulator pattern, where we incrementally accumulate values into a single variable to compute one feature of an array (e.g. sum or maximum).

  • Iterate over every pair 0≤i≤j<n0≤i≤j<n0≤i≤j<n﻿.
  • For each compute A[i]+A[i+1]+⋯+A[j]A[i] + A[i+1] + \dots + A[j]A[i]+A[i+1]+⋯+A[j]﻿.
  • Return the pair with the maximum value.
def naive algorithm(A):
	curr_val, curr_ans <- 0, (None, None)
	n <- len(A)
	for i in [0:n]:
		for j in [i:n]:
			aux <- sum(A[x] for x in [i:j+1])
			if aux > curr_val:
				curr_val, curr_ans <- aux, (i, j)

	return curr_ans

Note on Assignment Style

For assessments, you will have to design and analyze an algorithm for a given problem. This always consists of three steps:

  • Describe your algorithm: a high-level description in English, optionally followed by pseudocode.
  • Prove its correctness: a formal proof that the algorithm does what it’s supposed to do.
  • Analyze its time complexity: a formal proof that the algorithm runs in the time you claim it does.
